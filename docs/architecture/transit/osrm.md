# OSRM Travel-Time Matrix

> Architecture reference for the travel-time matrix generation pipeline.
> Source files: `apps/api/src/modules/transit/network/travel-time/osrm.service.ts` and `travel-time.controller.ts`

---

## Overview

The travel-time matrix stores pre-computed driving durations and distances between pairs of `TransitLocality` records. It is used throughout the system for dead-run scheduling, block scoring, and trip normalization.

Matrix entries are generated by calling the [OSRM Table API](http://project-osrm.org/docs/v5.5.0/api/#table-service) with all relevant localities in a single request, then persisting the N×N result. The generation runs asynchronously via `JobService`.

Entry point: `POST /transit/travel-time-matrix/generate` → spawns a job → calls `OsrmService.generateMatrix()`.

## Step 1 — Relevant Locality Selection

Only the subset of localities actually needed by the system is sent to OSRM, not the full `TransitLocality` table. A locality is included if it satisfies any of:

| Condition | Source |
|-----------|--------|
| Is a depot (`isDepot = true`) | Dead-run origin / destination |
| Is a route origin or destination | `TransitRoute.originLocalityId` / `destinationLocalityId` |
| Is a route waypoint | `RouteLocality.localityId` (used as `deltaMinutes` fallback) |

Localities that pass the relevance filter but have `lat` or `lng` set to `null` are silently skipped (`skipped` counter). If fewer than 2 localities have valid coordinates the request is aborted early.

---

## Step 2 — OSRM Request

### Endpoint

```
GET {OSRM_URL}/table/v1/driving/{coords}?annotations=duration,distance&radiuses={radiuses}
```

### Coordinate format

OSRM expects **longitude,latitude** (not lat,lng). Localities are serialized in the order they appear in the query result:

```
lng1,lat1;lng2,lat2;...;lngN,latN
```

### `radiuses` parameter

Every locality is sent with `radiuses=unlimited`. Without this, OSRM uses a default search radius (~15 m) and returns `null` for any point farther than that from a mapped road. `unlimited` ensures OSRM always snaps to the nearest road regardless of distance, eliminating silent null pairs caused by poorly georeferenced localities.

---

## Step 3 — Snap Info Persistence

The OSRM response includes a `sources` array containing one waypoint per input coordinate. Each waypoint reflects the location OSRM actually used after snapping to the road network:

```json
{
  "location": [lng, lat],
  "distance": 42.3,
  "name": "Rua das Flores"
}
```

`distance` is the gap in metres between the original input coordinate and the snapped road point. A value of `0` means the point was already on a mapped road.

After receiving the response, `generateMatrix()` persists a `snapInfo` JSON object on each `TransitLocality`:

```ts
snapInfo: {
  lat:       number,   // snapped latitude
  lng:       number,   // snapped longitude
  distanceM: number,   // metres from original coordinate to snapped point
  roadName?: string,   // name of the road the point was snapped to
  checkedAt: string,   // ISO 8601 timestamp of this run
}
```

`snapInfo` is updated on every matrix generation run, so it always reflects the most recent OSRM check. If one or more localities had `distanceM > 0`, a warning is logged with the count.

### Acting on snap data

`snapInfo` is a diagnostic field — it does not alter the stored `lat`/`lng` of the locality automatically. To correct a misaligned locality, an explicit action must copy `snapInfo.lat`/`snapInfo.lng` into the locality's `lat`/`lng` fields and clear `snapInfo`. Re-running the matrix generation after that update will produce entries based on the corrected coordinates.

## Step 4 — Matrix Update

### Unit conversions

| OSRM field | Unit   | Stored as    | Conversion                          |
|------------|--------|--------------|-------------------------------------|
| `duration` | seconds | `baseMinutes` | `Math.ceil(seconds / 60)`          |
| `distance` | metres  | `distanceKm`  | `Math.round(metres / 10) / 100`    |

### Manual entry protection

Before building the insert payload, all `TravelTimeMatrix` rows with `source = 'MANUAL'` are loaded into a Set. Any pair that appears in this Set is skipped — manual overrides are never overwritten by OSRM.

### Null pairs

OSRM returns `null` in the duration/distance matrices when it cannot find a driving route between two points. These pairs are counted (`nullPairs`) and skipped. Even with `radiuses=unlimited` this can happen for islands or localities in road-network-isolated areas.

### Atomic replace

The final write uses a single Prisma transaction:

```
1. travelTimeMatrix.deleteMany  — remove all OSRM entries for the locality set
2. travelTimeMatrix.createMany  — insert the freshly computed entries
```

This scoped delete ensures that OSRM entries for localities outside the current run (e.g. localities removed from all routes) are not affected until they are included in a future run.

---

## Async Execution

The controller creates a `Job` record before calling `generateMatrix()` and returns its `jobId` immediately. The generation runs in the background via `JobService.run()`. Clients poll the job endpoint to track progress.

```
POST /transit/travel-time-matrix/generate
→ { jobId: "..." }
```

## TravelTimeMatrix Schema

| Field          | Type    | Description                                    |
|----------------|---------|------------------------------------------------|
| `originId`     | uuid    | FK → `TransitLocality`                         |
| `destinationId`| uuid    | FK → `TransitLocality`                         |
| `baseMinutes`  | int     | Driving time in minutes (ceiling of OSRM seconds) |
| `distanceKm`   | float   | Driving distance in kilometres                 |
| `speedRatio`   | float   | Multiplier applied at consumption time (default 1.0) |
| `source`       | enum    | `OSRM` (auto-generated) or `MANUAL` (user override) |
